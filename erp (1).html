Security Architecture
Defense-in-depth — edge to database
Security layers (outer → inner)
Layer 1 — EdgeCloudflare / CloudFront
DDoS protection (L3/L4/L7) · CDN WAF with OWASP ruleset · TLS 1.3 only, HSTS enforced · Bot detection · Rate limiting at edge (1K req/min standard, 10K enterprise)
Layer 2 — Identity & AuthKeycloak
OAuth 2.0 + OIDC + SAML · MFA mandatory for financial/admin roles (TOTP + SMS OTP) · JWT tokens (8h web, 30d mobile) · Refresh token rotation on every use · Concurrent session control · Geo-fencing (optional) · IP allowlisting for Accountant/Auditor · 5-attempt lockout → 15min freeze
Layer 3 — API GatewayKong
JWT validation on every request · Rate limiting per tenant and per route · Request size limits · SQL injection prevention via input sanitization · API key management (scoped keys for POS/webhook integrations)
Layer 4 — ApplicationRBAC + ABAC
15-level RBAC hierarchy · Permission check on every API handler · Parameterised queries (ORM-enforced, zero raw SQL) · CSRF tokens on all state-changing requests · CSP headers · XSS prevention via output encoding · No PII in logs (redacted at write time)
Layer 5 — DataEncryption + RLS
AES-256 at rest for all stores incl. SQLite offline cache · TLS 1.3 in transit · PCI DSS tokenisation (no raw card data stored) · P2PE terminal support · Row-Level Security per tenant_id · Vault-managed dynamic DB credentials · Annual pen test + daily vulnerability scan
Compliance & audit
Certifications
  • PCI DSS SAQ-A/SAQ-B (card payments)
  • GDPR / DPDPA compliance (data minimisation, erasure)
  • ISO 27001 roadmap (Enterprise tier)
  • SOC 2 Type II roadmap
  • GSTN integration compliance (India)
Audit log
  • Immutable, tamper-evident log of every action
  • Fields: user_id, tenant_id, action, resource_id, old_val, new_val, IP, device, timestamp
  • Stored in append-only Postgres + S3 archive
  • Searchable by Auditor role (read-only)
Secret management
  • HashiCorp Vault for all secrets
  • Dynamic DB credentials (auto-rotated per pod)
  • AWS Secrets Manager for cloud-native services
  • Zero hardcoded credentials anywhere in codebase
  • Automated secret rotation (30-day cycle)

Integration patterns
Webhook design (outbound)
  • HMAC-SHA256 signed payloads (X-Platform-Signature header)
  • Retry policy: 3 attempts at 1min → 5min → 30min
  • Idempotency key on every event (UUID)
  • Events: transaction.completed, inventory.low_stock, delivery.status_changed, payment.failed, fuel.tank_low
  • Webhook health dashboard for each endpoint
API key management
  • API keys scoped to specific endpoints (not full access)
  • Hash of UUID stored; original shown once on creation
  • Per-key rate limits configured in Kong
  • Key rotation without downtime (grace period on old key)
  • Keys attributed to integration name + creation timestamp in audit log

AI governance & trust framework
Transparency rules
  • All outputs labeled "AI suggestion" — never presented as system rule
  • Every auto-action logged with model version, confidence score, features used
  • SQL shown alongside NL query results (user can verify)
  • Explainability required on all customer-facing outputs
Data governance
  • Zero cross-tenant data in training — models trained on own tenant data only
  • No PII in model feature vectors
  • Bias monitoring: fraud + churn models audited quarterly for demographic/time bias
  • Users rate suggestions (👍/👎) — feedback fed to retraining pipeline
Output modes
  • Alert — immediate attention, anomaly detected
  • Suggestion — human confirms before action taken
  • Auto-action — low-risk reversible only (e.g. PO draft, not send)
  • Insight — informational, no action required

k6 automated tests (CI-embedded)
